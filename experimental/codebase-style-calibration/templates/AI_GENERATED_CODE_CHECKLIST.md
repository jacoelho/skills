# AI-Generated Code Checklist

Use this checklist when reviewing code generated by an AI agent for this repository.

## Immediate rejection signals

- [ ] Introduces abstractions not required by the current change.
- [ ] Adds wrappers that only forward arguments and return values.
- [ ] Adds generic `Manager`, `Service`, `Provider`, `Factory`, or `Handler` types without a precise domain role.
- [ ] Adds defensive nil checks where constructors or invariants should make nil impossible.
- [ ] Adds comments that restate obvious code.
- [ ] Adds helpers used once without naming a real concept.
- [ ] Adds broad configurability before there is a real second use case.
- [ ] Adds tests that assert implementation detail instead of behaviour.

## Required review questions

- [ ] Does this code look native to the repository?
- [ ] Is every new abstraction earning its cost?
- [ ] Could fewer concepts express the same behaviour?
- [ ] Are invalid states made impossible where practical?
- [ ] Are errors useful to the caller, not just readable to a human?
- [ ] Do tests protect behaviour during refactoring?
- [ ] Does the change preserve hot-path assumptions?
- [ ] Would this survive review if written by a human contributor?
